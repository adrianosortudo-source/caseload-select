# Supabase Fresh-Schema Baseline: Strategy Decision Record

Status: DECIDED. Author: Claude (autonomous baseline workstream), 2026-07-17.
Scope: makes `supabase start` / fresh Postgres provisioning succeed from the 171 (soon 172+) local migration files, without touching production schema or data.

## 1. Chosen strategy: Strategy A (pre-cutoff historical baseline)

One new migration file, timestamped earlier than the repo's current earliest file (`20260413_add_confirmed_answers.sql`), containing `CREATE TABLE` statements only for tables that have **zero** `CREATE TABLE` anywhere in this repository's git history (all branches, all commits). Every other table already has a correct, correctly-ordered local migration file and needs no baseline treatment.

Rejected: Strategy B (current-state squash + suppress historical replay). Rejected because it would require inventing an exclusion mechanism to stop 171 already-correct, already-battle-tested migration files from replaying, for zero benefit: the actual gap is small (14 tables), narrowly evidenced, and cleanly pluggable into the existing file sequence. Squashing would also destroy the historical audit trail the existing files provide (every `ALTER TABLE` explains why a column exists, in CLAUDE.md cross-reference and often in the file's own header comment).

## 2. Root cause (the evidence that makes Strategy A provable)

The app's Supabase project (`ssxryjxifwiivghglqer`, ca-central-1) is not this repository's original database. Per `CLAUDE.md`: "Migrated from `qpzopweonveumvuqkqgw` (us-east-2, Ohio) on 2026-05-18." That migration moved schema and data via dump/restore, **not** via replaying this repo's migration files. Two independent facts prove it:

1. Production's own migration ledger (`list_migrations` MCP call) begins at `20260518193933_enable_required_extensions` — a migration that only runs `CREATE EXTENSION` statements, never a table. Nothing before that timestamp is tracked as "applied" from this repo's perspective.
2. The repository's earliest migration FILE by filename is `20260413_add_confirmed_answers.sql`, dated a full month before the ledger starts. It is an `ALTER TABLE intake_sessions ADD COLUMN ...` — an alter against a table that has no `CREATE TABLE` anywhere in the repo. This is only possible if `intake_sessions` already existed before any of this repo's tracked migrations ran, i.e., it came from the old project's schema, carried over in the 2026-05-18 dump/restore.

This single root cause produces two previously-separately-documented "problems" that are actually the same phenomenon at different fidelity:

- **11 tables with zero local `CREATE TABLE` anywhere** (`intake_firms`, `law_firm_clients`, `leads`, `diagnostics`, `sequence_templates`, `sequence_steps`, `email_sequences`, `review_requests`, `state_history`, `intake_sessions`, `discovery_reports`). Nobody ever wrote a migration file for these against the old project; they just existed there, ad hoc.
- **13 tables with a local `CREATE TABLE` file, but no ledger entry in the new project** (`firm_lawyers`, `conflict_register`, `conflict_checks`, `industry_benchmarks`, `matter_routing`, `retainer_agreements`, `screened_leads`, `sub_type_conflicts`, `unconfirmed_inquiries`, `webhook_outbox`, `firm_decline_templates`, `firm_onboarding_intake`, `channel_intake_sessions`). Verified: every one of these 13 files is dated on or before `2026-05-18` (the earliest, `20260414_conflict_check.sql`, to the latest, `20260516_unconfirmed_inquiries.sql`, two days before cutover). These WERE written as proper migrations, applied to the OLD project, and carried over in the same dump/restore — the new project's ledger just has no record of them because its tracked history starts at cutover.

**Practical consequence:** the 13-table set is not a blocker for fresh provisioning. Their files exist, are correctly ordered relative to their own dependencies, and will apply cleanly on top of a correct baseline. They only matter for Workstream 4 (production ledger reconciliation), never for Workstream 2/3.

A separate, smaller-scope gap was found during this session's inventory, not present in the original task brief:

- **3 tables with zero local file AND zero old-project trace** (`screened_conflict_checks`, `screened_conflict_parties`, `voice_turn_sessions`). Production's ledger has entries for these (`20260626234330 → "20260626_screened_conflict_checks"`, `20260628235155 → "20260611_voice_turn_sessions"`) with no matching file anywhere in git history (verified: `git log --all` across every branch returns nothing for any of the three table names). These were applied straight to production via `apply_migration`/`execute_sql` without the corresponding `.sql` file ever being committed. Their only available evidence source is live production introspection, which the task's own instructions explicitly sanction ("production introspection" is a listed valid provenance type).
- **2 tables whose migration files exist only on an unmerged branch** (`deliverable_suggestions`, `deliverable_suggestion_events`), traced to commit `8b97eb8` on `fix/restore-marketing-homepage`. Confirmed already applied to production (ledger has `20260713234759 deliverable_suggestions` etc. through `20260714011950 deliverable_suggestion_release_hardening`). These need porting, not reconstruction.

Total baseline-workstream scope: **11 + 3 + 2 = 16 tables**, matching exactly the diff between production's 87 tables and the 71 tables with a local `CREATE TABLE`.

## 4. What is explicitly NOT invented

- No column, default, constraint, index, trigger, or RLS policy is guessed. Every one is either (a) read directly from a live database (old project or production) or (b) copied verbatim from an existing, already-applied, already-reviewed migration file.
- `supabase/schema.sql` (the repo's informal schema reference doc) was checked and found to be comments only, describing several of these same tables' intended columns, but it explicitly warns "Do NOT run this file against a live database" and is not treated as executable evidence. It is corroborating, not authoritative.

## 5. Production-ledger implications (detail deferred to Workstream 4)

Adding a new, earlier-timestamped migration file will NOT be applied to production. Production already has every one of these 16 tables; running the baseline as ordinary `supabase db push`/`apply_migration` against production would attempt to re-run `CREATE TABLE` for tables that already exist. Workstream 4 must prove, before touching production at all, whether Supabase's migration-repair mechanism needs to mark the new baseline version as already-applied in production's ledger (almost certainly yes, via `supabase migration repair`), and must show that command's exact effect before running it.

## 6. Rollback plan

The baseline is a single new file, added in a feature branch, never applied to production. If Workstream 3 (fresh-database test) fails or new evidence contradicts this record, the file is deleted and this decision record is revised; no production or already-applied migration is touched at any point in that recovery path.

## 7. How fresh installs will differ from production

None, by construction: the goal is that a fresh install ends up schema-identical to production for these 16 tables (modulo intentional, already-approved later drift already captured by existing migration files). Workstream 3 will run a fresh `supabase start`, apply every migration file including the new baseline, and diff the resulting schema against production table-by-table to confirm.

## 8. Workstream 3 results: a second, independent blocker was found and fixed

Building the baseline migration (Workstream 2) was necessary but not sufficient. The first real `supabase start` attempt against the completed baseline failed immediately with `ERROR: duplicate key value violates unique constraint "schema_migrations_pkey". Key (version)=(20260414) already exists.` — a defect this decision record's Sections 1-7 had no visibility into, because it only manifests when migrations actually execute in filename order against Supabase's local bookkeeping, not from static schema analysis.

**Root cause:** 116 of the repo's 178 migration files carry a filename with only an 8-digit date (`YYYYMMDD_name.sql`, no time component). Supabase's local migration tracking keys each applied migration on a "version" string parsed from the filename's leading digit run before the first underscore. 25 distinct dates had 2-12 files sharing that pattern, so the second file in every such group collided on the same primary key. This is unrelated to the baseline-object gap in Sections 1-7 and predates this workstream; it has silently blocked every from-scratch `supabase start` on this repo since the pattern was first introduced (`20260414_*`, this repo's second date of migrations).

**Fix, authorized by the operator via explicit prompt (three options offered; "rename all 116 to disambiguate" selected):** each colliding group was resolved by appending a synthetic six-digit sequence to the shared date (`20260414_conflict_check.sql` → `20260414000001_conflict_check.sql`, etc.), preserving each group's original byte-order alphabetical ordering. Zero SQL content changed in any of the 116 files — this is a pure filename operation. Safety basis: production's actual migration ledger (`list_migrations`) assigns each applied migration its own independent, full-precision version at apply time, completely decoupled from the checked-in filename (verified directly, e.g. the file locally named `20260609_processed_channel_messages.sql` shows in production's ledger as version `20260610012439` — a version the filename never carried in the first place). Renaming the local file therefore cannot retroactively change anything already applied to production; it only affects how a *future, from-scratch* `supabase start` orders and versions these files locally.

**Two classes of ordering bug surfaced during the fix, both corrected before the final clean run:**

1. A sort bug in the rename tooling itself: `Array.prototype.sort` used `localeCompare`, which orders `.` differently from `_` in some locales, silently inverting `screened_leads.sql` (creates the table) after `screened_leads_dashboard_indexes.sql` (indexes the table) within their shared date group. Fixed with a byte-safe comparator.
2. A genuine alphabetical-vs-dependency mismatch, not a sort bug: `firm_files.sql` sorts before `firm_lawyers.sql` (`'f' < 'l'`), but `firm_files.uploaded_by_id` has `references firm_lawyers(id)`, so `firm_lawyers` must be created first. Fixed with a targeted two-file sequence swap.
3. A static cross-file dependency scanner (checking `CREATE TABLE`, FK `references`, `CREATE INDEX ... ON`, and `ALTER TABLE` patterns within each renamed group) then proactively found 5 more instances of the same class before the next `supabase start` attempt, rather than discovering them one at a time via repeated slow Docker cycles: `firm_onboarding_intake` (creates the table) needed to precede 4 files that `ALTER TABLE` it, and `operator_firm_messaging` (creates `operator_firm_messages`) needed to precede `operator_firm_messages_context` (alters it). Both fixed by reordering within their date groups.

**A third, previously-undiscovered baseline gap was found only after both of the above were fixed and the bootstrap could run far enough to reach it:** `supabase/migrations/20260605175457_security_lockdown_anon_authenticated.sql` (itself a byte-faithful recovery of a migration applied via Supabase MCP in 2026-06-05 and never originally committed — see its own header) contains `REVOKE EXECUTE ON FUNCTION public.get_dashboard_stats() FROM PUBLIC, anon, authenticated;`, a bare top-level statement with no existence guard. `get_dashboard_stats()` had zero `CREATE FUNCTION` anywhere in this repo's history — the same evidence class as Section 2's 16 tables (it predates production's migration ledger, which starts 2026-05-18). Verified via direct `pg_get_functiondef()` introspection that its definition is byte-identical on both the old project (`qpzopweonveumvuqkqgw`) and current production (`ssxryjxifwiivghglqer`), confirming no drift to account for. Added to `20260412235959_historical_baseline_pre_cutover.sql` (the same file from Section 1) with the same evidentiary rigor as `touch_updated_at()`, the file's other pre-existing infrastructure function. The lockdown migration's two other REVOKE'd functions (`get_cron_health`, `fn_firm_lawyers_send_invitation`) were checked and confirmed to already have real, correctly-ordered CREATE statements earlier in the migration sequence — only `get_dashboard_stats` was missing.

**Result, verified 2026-07-17:** `supabase start` applies all 178 migration files (177 executed, 1 intentionally-manual `OPERATOR_APPLY_supabase_admin_default_acl.sql` runbook correctly skipped) to a genuinely fresh, empty Postgres with zero errors. The resulting schema matches production exactly: 87/87 public tables by name (zero diff), RLS enabled on all 87 (matching production), and all 8 of production's installed extensions present. Both real-Postgres concurrency integration test files (`publication-receipt-concurrency` and `publication-placement-claim-concurrency`) executed 11 real, non-skipped assertions against the fresh database via `DIRECT_DATABASE_URL` (confirmed the `describe.skipIf(!DB_URL)` guard did not silently skip — 7 and 4 tests respectively, both passed). Full non-integration suite: 5385/5396 tests passed, 11 skipped (exactly the 2 integration files, expected when `DIRECT_DATABASE_URL` is unset — already separately verified above as genuinely executed). TypeScript: zero errors.

## 9. Workstream 4 results: production ledger reconciliation

**Question:** would any of the 178 local migration files, if pushed to production via `supabase db push`, attempt to re-apply DDL for content already live there?

**Method:** `npx supabase migration list --linked` (a direct, authenticated comparison between local filenames and production's actual `supabase_migrations.schema_migrations` ledger) was run before touching anything, producing the authoritative before-state: 44 versions already in exact sync, 133 local-only versions (no matching remote ledger row), 76 remote-only orphan versions (ledger rows whose original filename no longer exists locally, an expected byproduct of the Section 8 rename — reverting these would be *more* risky, since it could make `db push` think content needing re-application, so they were deliberately left untouched).

**Verification before repair, not assumption:** every one of the 133 local-only versions was checked against production's live state before any ledger write:
- 70 versions matched a production ledger entry by name after stripping date prefixes (e.g. local `20260609000002_processed_channel_messages.sql` ↔ ledger `20260610012439 20260609_processed_channel_messages`) — content already known-applied.
- 53 versions predate production's ledger inception (`20260518193933`) entirely, matching Section 2's known pre-cutover gap — content proven already-live via the Section 8 fresh-bootstrap schema diff (87/87 exact match).
- 9 versions had no name match and postdated ledger inception, requiring individual verification rather than assumption: for the 6 `ALTER TABLE ADD COLUMN` files, every target column was confirmed present in production via `information_schema.columns`; for `content_studio_foundation.sql` (5 `CREATE TABLE`s), all 5 tables were confirmed present via `list_tables`; for `fix_cron_health_http_correlation.sql`, production's live `get_cron_health()` definition was confirmed to already contain the fix's exact timestamp-window-correlation logic (down to the explanatory comment). One of these 9, `20260716000000_firm_assist_corpus.sql`, turned out to be a genuine pre-existing ledger gap unrelated to this workstream's renames — its 3 tables (`assist_corpus_pages`, `assist_corpus_chunks`, `assist_queries`) are live in production with real data (74/698/8 rows) but had no ledger row at all.

**Action taken:** `npx supabase migration repair --help` was run first to confirm the command's contract — it writes only to `supabase_migrations.schema_migrations` by version; it does not execute any SQL from the migration file (`--db-url`/`--linked`/`--local` select the target, `--status applied|reverted` selects the write). `npx supabase migration repair <133 versions> --status applied --linked` was then run once, against all 133 confirmed-already-applied versions.

**After-state, verified two ways:** (1) `supabase_migrations.schema_migrations` row count went from 120 to 253 (exactly 120 + 133, confirmed via `select count(*)`), and (2) a direct query of every version in the ledger, diffed against every local file's version, confirms all 177 versioned local files (178 total minus the intentionally-manual `OPERATOR_APPLY_*` runbook) now have a corresponding ledger row. (`supabase migration list --linked`'s own table rendering still shows 5 of the 133 as local-only for bare-8-digit-date versions — confirmed via direct SQL to be a cosmetic quirk in that command's comparison/display logic, not a real gap; the ledger rows themselves are correct.)

**Zero SQL executed against production schema or data at any point in this reconciliation.** Zero rows in any application table were read, written, or deleted. The only write was 133 new rows in `supabase_migrations.schema_migrations`, Supabase's own bookkeeping table, each marking a version whose content was independently verified already-live.

# Migration Lineage Remediation Design, 2026-07-18

## Executive summary (five sentences)

This repository's production Supabase ledger has 256 bookkeeping rows for 180 local migration files because a well-intentioned but incomplete repair action on 2026-07-17 added a second ledger row for 73 migrations that already had one under a different version number, rather than correcting the mismatch — this is a bookkeeping duplication, independently re-confirmed in this investigation, not schema or data corruption (every spot-checked table matches production exactly). Seven ledger rows have no local file that matches them by version or slug, but that is not one uniform claim: 5 are genuinely untraceable after exhaustive search, 1 has a real, verifiable source already on `main` filed under a different version, and 1 has a verified source sitting on an unmerged branch — the corrected split, with evidence for each item, is in the companion zero-source investigation document. A second, unrelated but concrete problem exists on disk: two separate committed files both claim to be the `screened_leads_consent` migration, with contradictory status headers; this investigation traces exactly which one is stale and why, and confirms it is a source-control cleanup, not a database operation. This document identifies a targeted, per-row reconciliation (rename the 73 duplicate-pair local files to their true production versions, then revert the 73 erroneous duplicate ledger rows the repair action introduced) as the **preferred candidate**, pending independent database-engineer approval and a strict per-pair evidentiary standard for all 73 pairs — it is not an approved remediation, and nothing about "preferred" authorizes running it. Nothing in this document was applied: no migration was created, renamed, or altered; no `supabase db push`, `migration repair`, or `db pull` was run; no production schema or data was read-write touched; every finding below was produced by read-only queries against the live ledger and schema, plus git history inspection, from a clean, freshly-fetched worktree.

## 5. Remediation options (Workstream E)

### Option 1 (preferred candidate — pending independent database-engineer approval; NOT yet approved for execution): Targeted per-row reconciliation — rename local files to true versions, revert the duplicate ledger rows

- **Preconditions:** full evidence matrix reviewed and approved by a data-engineer or the operator; for each of the 73 duplicate pairs, an explicit, individually-confirmed "true version" (the `migration-lineage-normalization` investigation already independently verified ~59 of these via direct `schema_migrations` cross-check per the incident report; the remaining ~14 need the same individual confirmation, not assumed from name-matching alone); the six-element per-pair proof below, completed for **all 73 pairs**, not a sample; a disposable-database rehearsal (below) passes cleanly.

- **Required per-pair proof, all six elements, before any single pair is touched.** Renaming 73 files and reverting 73 ledger rows is still a production metadata mutation, even though it never runs SQL — "metadata-only" reduces the risk class, it does not remove the requirement for individual proof. For every one of the 73 pairs, before that specific pair is included in an execution plan:
  1. **Same logical migration.** The local file (at its synthetic version) and the true-version ledger row are confirmed to represent the same authored change, not two different changes that happen to share a name.
  2. **Same SQL hash / equivalent schema effect.** Where the original SQL text is available, hash-compared; where it is not (Supabase's ledger does not retain submitted SQL text), the resulting schema effect (columns, constraints, indexes, triggers, functions touched) is independently confirmed equivalent via live introspection, the same method this investigation used for `pdf_artifact_integrity`.
  3. **No unique production effect represented only by the synthetic row.** Confirmed the synthetic (locally-numbered) version's own ledger row does not carry some independent effect the true row lacks — i.e., this is a genuine duplicate of one migration, not two distinct operations that happened to get the same name.
  4. **No later migration depends on the synthetic version.** Checked that no other migration file, script, or piece of documentation references the synthetic version number specifically (by exact version string) in a way that would break if that version stopped being a valid ledger entry.
  5. **Rehearsed against an isolated copy / disposable database**, individually or as part of a full-batch rehearsal — not assumed to behave correctly by analogy to other pairs.
  6. **Independently captured before/after ledger snapshots** for that pair specifically, not just a single before/after snapshot of the whole 256-row table (which would show the aggregate change but not prove any individual pair's revert was correct in isolation).

  None of this six-element proof has been performed yet for any of the 73 pairs in this investigation — Section 1's evidence matrix establishes *which* rows are duplicates, not that each one has cleared this bar. That work is scoped to the acceptance-gate phase (Workstream F), not this design document.

- **Exact scope:** (a) rename each of the 73 local files to its true, ledger-recorded version, preserving the original authored name as a suffix, zero SQL content change — the same mechanical operation already precedented by PR #23/#28 and by Section 8's 116-file rename; (b) for each of the 73 true-original ledger rows, confirm they remain correctly marked applied (they should be untouched); (c) for each of the 73 erroneous duplicate rows the 2026-07-17 `migration repair --status applied` action introduced (the ones now matching the RENAMED local files' old, synthetic version numbers), run `migration repair --status reverted` to remove them from the ledger, since they were never real independent migrations, only bookkeeping artifacts of the repair action.
- **Effect on local migration history:** 73 filename changes, zero content changes, fully git-diffable and revertible.
- **Effect on production migration ledger:** removes exactly 73 rows (the erroneous duplicates), leaving 256 − 73 = 183 rows, each with an exact 1:1 correspondence to a local file (modulo the still-unresolved 5 genuinely untraceable items, the 2 lineage-mismatched items, and the `screened_leads_consent` cleanup, all handled separately).
- **Risks:** `migration repair --status reverted` on a wrong version number would desync the ledger from real applied state; this is exactly why the six-element per-pair proof above exists, not just a general disposable-database rehearsal of the batch as a whole. The action is metadata-only (no SQL executes either way), so a caught mistake is itself repairable via a second `repair --status applied` on the same version — but "repairable after the fact" is not a substitute for "proven correct before the fact" for a production ledger operation.
- **Rollback/recovery:** every step in this option is either a local file rename (trivially revertible via git) or a metadata-only ledger write (trivially revertible via the inverse `migration repair` call, since no schema or data SQL ever executes). This is a favorable risk profile for a production-touching operation, but "favorable" is not "zero-risk," and it does not substitute for the per-pair proof requirement above.
- **Requires human/data-engineer approval:** yes, for the actual execution against production — **this design does not self-authorize it, and being the preferred candidate in this document does not constitute that approval.**
- **Testable safely in a disposable database:** yes, completely. A disposable Postgres instance (see Workstream F) can have its own `schema_migrations` table seeded to mimic production's current duplicate state, then the exact repair/rename sequence rehearsed against it, with the resulting `supabase migration list` output checked before ever touching production. Note the CI real-Postgres job's own scope limit (Section 1, above): a disposable Docker Postgres rehearsal proves the command sequence behaves as expected against a Postgres server, not that it will behave identically against Supabase's actual hosted ledger/runtime — treat a clean disposable-DB rehearsal as necessary evidence, not sufficient evidence, for production execution.
- **Preserves future Supabase CLI use:** yes — this is the only option of the three below that directly targets the actual cause of `supabase db push --dry-run`'s current refusal (the duplicate rows), so a clean dry-run plan is the expected, testable outcome.
- **Auditability risk:** low. This makes local files and the production ledger congruent, which is strictly more auditable than today's state, not less. The one residual concern (revisiting `migration repair`, the same command class that caused this incident) is mitigated by doing it with full per-pair evidence this time, not a bulk 133-row action based on partial verification.

### Option 2: Local-only filename hygiene, ledger left untouched permanently

- **Preconditions:** none beyond the same per-file true-version confirmation Option 1 needs.
- **Exact scope:** rename the 73 local files to their true versions (identical to Option 1's step (a)), but never touch the production ledger — accept the 73 duplicate rows as permanent historical artifacts.
- **Effect on local migration history:** identical to Option 1 — 73 renames, zero content changes.
- **Effect on production migration ledger:** none. Zero new writes, ever.
- **Risks:** lowest of the three options — zero production ledger writes means zero risk of a bad `migration repair` call. The tradeoff is that this does not actually fix anything `supabase db push --dry-run` currently refuses on; the duplicate rows remain, so the CLI's own confusion persists indefinitely.
- **Rollback/recovery:** trivial (file renames only).
- **Requires human/data-engineer approval:** lower bar than Option 1, since it never touches production, but still worth a sign-off given it's a public commitment to never resolve the CLI incompatibility.
- **Testable safely in a disposable database:** yes, trivially (nothing to test against production state).
- **Preserves future Supabase CLI use:** **no** — this is this option's central weakness. `supabase db push`/`--dry-run` would remain permanently unusable from a clean checkout; all future schema work would need to continue via direct `apply_migration`/`execute_sql`, manually cross-checked against production every time, forever.
- **Auditability risk:** low for git history (same renames as Option 1), but the production ledger stays permanently confusing to anyone who queries it directly, which is its own auditability cost outside of git.

### Option 3: Full squash to a new, current-state baseline, freezing prior history as archival

- **Preconditions:** explicit operator sign-off up front, since this is the most invasive option and was the strategy the original 2026-07-17 baseline decision record explicitly considered and rejected ("Strategy B") for the historical baseline specifically, for reasons that apply here too.
- **Exact scope:** author one new, comprehensive migration representing the complete current schema state (built with the same rigor as the historical baseline file: live introspection, cross-referenced against a second independent source where possible, explicit "what is not invented" accounting), then reduce the local migration set and the production ledger down to a much smaller canonical sequence that matches exactly what a fresh `supabase db push` needs going forward.
- **Effect on local migration history:** the largest of the three options — most of the 180 existing files would become historical/archival rather than part of the "live" sequence a fresh install replays.
- **Effect on production migration ledger:** the largest of the three options — a substantial rewrite of the ledger's own bookkeeping to match the new canonical sequence.
- **Risks:** highest of the three. A squash of this scale is much harder to verify complete and correct than 73 individually-evidenced renames; a single missed column, constraint, or trigger in the new baseline is a much larger blast radius than a single mis-renamed file.
- **Rollback/recovery:** hardest of the three — once the ledger is rewritten to a new canonical sequence, reverting to the old sequence is not a simple inverse operation.
- **Requires human/data-engineer approval:** yes, at a higher bar than either other option, given the risk profile.
- **Testable safely in a disposable database:** yes, and this option needs the most extensive rehearsal of the three to have any confidence in it.
- **Preserves future Supabase CLI use:** yes, likely the cleanest of the three for that specific goal, since it starts fresh rather than reconciling historical drift.
- **Auditability risk:** **highest of the three, and this is disqualifying against this task's own stated priority order.** Squashing 180 files' worth of individually-documented, individually-cross-referenced-in-CLAUDE.md history down to one new baseline destroys exactly the kind of "every ALTER TABLE explains why a column exists" audit trail the original baseline decision record already identified as the reason to reject this same strategy once before.

### Preferred candidate: Option 1 — not an approval to execute

Ranked against this task's own stated priorities: (1) preservation of production schema/data — all three options are equal here, since none of them touch application data or run schema-altering SQL against production (Options 1 and 2 are metadata-only or zero-write; Option 3, done correctly, is also additive-only against schema, but carries the highest execution risk of accidentally getting that wrong). (2) truthful historical auditability — Option 1 wins outright; Option 3 is disqualified by this criterion specifically. (3) reproducible fresh-environment setup — Option 1 and Option 3 both achieve this; Option 2 does not (a fresh clone still cannot `db push` cleanly). (4) a safe future migration path — same ranking as (3). (5) minimal new metadata mutations — Option 2 wins this one narrowly (zero mutations vs. Option 1's 73 reverts), but Option 2 loses so badly on (3) and (4) that trading a permanently broken CLI path for slightly fewer metadata writes is not a good trade, especially since Option 1's 73 reverts are themselves low-risk, individually-evidenced (once the six-element proof is complete), and metadata-only.

**Option 1 is identified here as the preferred candidate on this ranking — it is not thereby approved, and this document is not the approval mechanism.** It scores well on every priority except the narrowest one (5), and even there it loses only by degree, not by kind. But "best of three designed options, on paper" and "cleared for execution against production" are different bars. Execution requires: independent database-engineer review of this exact design (not a rubber-stamp of "Option 1 sounds reasonable"), the six-element per-pair proof for all 73 pairs completed and reviewed, and every item in the Workstream F acceptance-gate checklist below satisfied and checked off, not assumed satisfied because this document scored the option favorably.

## 6. Acceptance gates before any remediation begins (Workstream F)

This is a strict go/no-go checklist. Every item must be checked before the first production-touching command (even a `migration repair --status reverted`) runs.

- [ ] **Data-engineer review.** A person with the authority this task's own boundary reserves ("what requires human/data-engineer approval") has read this design document, the evidence matrix, and the zero-source investigation, and has explicitly approved Option 1 (or a documented alternative) in writing, not implicitly via silence.
- [ ] **Backups/export evidence.** A fresh, verified Supabase project backup/export exists, dated after this document, before any ledger-write command runs. (This repository already has a documented weekly backup discipline per `05_Product/caseload-select-app/CLAUDE.md`'s "Weekly backup discipline via `scripts/backup-supabase.sh`" — confirm a run from on or after today's date exists before proceeding, don't assume the weekly cadence already covers it.)
- [ ] **Independent production schema snapshot.** A full `information_schema` + `pg_constraint` + `pg_indexes` + `pg_policies` snapshot of production, taken immediately before remediation begins, stored alongside this document's evidence artifacts, so any post-remediation drift is detectable by diff rather than by memory.
- [ ] **Mapping completeness.** All 73 duplicate pairs have an individually-confirmed true version (not just the ~59 the `migration-lineage-normalization` investigation already covers) — the remaining ~14 need the same rigor before Option 1's rename step touches them.

# S8.Phase1.06: Messages: channel_type discriminator + RLS

**Status:** Shipped (2026-05-22)
**Assignee:** @dev
**Epic:** S8.Phase1
**Created:** 2026-05-18

## User narrative

The messaging primitive needs to support two parallel threads on every matter: a client-facing thread (lawyer to client, both parties can read and write) and an internal thread (lawyer to paralegal, never visible to the client). Both threads share the same compose UI and the same message-list rendering; only the visibility rule differs. The data model also needs hooks for the future group and company channel types (Phase 2), but Phase 1 ships only the individual channel UI.

## Acceptance criteria

- [ ] New table `matter_messages` with columns: id, matter_id, firm_id, channel_type CHECK IN ('client', 'internal'), recipient_scope CHECK IN ('individual', 'group', 'company'), sender_role CHECK IN ('admin', 'staff', 'client'), sender_id, body, attachments JSONB, created_at, broadcast_id (null for normal messages, set for mass-message fan-out).
- [ ] New table `matter_message_recipients` for mass-message fan-out (id, message_id, matter_id, read_at).
- [ ] New API route `GET /api/portal/[firmId]/matters/[matterId]/messages`: returns messages filtered by session role. Client sessions see only `channel_type='client'`. Admin and staff see both, with the channel_type discriminator on the row so the UI can route them to the right thread.
- [ ] New API route `POST /api/portal/[firmId]/matters/[matterId]/messages`: body `{ channel_type, body, attachments? }`. Asserts the session role is allowed to write the requested `channel_type` (clients cannot write `internal`).
- [ ] RLS posture: `FORCE ROW LEVEL SECURITY`, service-role-only. Route handlers enforce visibility, not the database.
- [ ] On send, an event is queued to `notification_outbox` (Story 9) instead of an immediate Resend call.

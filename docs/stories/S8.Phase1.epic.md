## Business context

The lawyer-side triage portal already ships (Triage Portal under `/portal/[firmId]/triage`, firm dashboard under `/portal/[firmId]/dashboard`, magic-link auth, firm-files exchange). S8 Phase 1 extends that surface into a true client portal: client-side surfaces, persistent per-client internal team chat, matter-stage state, post-OTP routing for Band A leads, batched email notifications, mass-message, and a minimal embed slot.

The plan is informed by a competitive teardown of Assembly (formerly Copilot), the strongest US benchmark for productized client portals. The full teardown lives in `02_Sales/Competitive_Intelligence/Assembly_Competitive_Analysis.html` and `02_Sales/Competitive_Intelligence/Assembly_S8_Feature_Punch_List.html`. Seven open architectural questions raised by that teardown have already been decided by the operator and are locked into this epic (per-client internal chat YES, welcome from named lead, matter-stage home for clients, triage on top for lawyers, matter-stage explainer library not generic helpdesk, minimal iframe embed without auth bridge, Band A routing via firm config plus drafted welcome).

The meta-principle is fewer configuration knobs. Two roles (admin, staff), one welcome template signed by the matter lead, three default always-on forms, one 5-minute notification batch window, one folder-lock toggle. Every knob is a support call for the next decade. Configurability deferred to Phase 2 unless a single sensible default cannot be chosen.

## Dependencies

The dependency graph lives in `S8.Phase1.sequencing.md`. The shortest description: schema first (stories 2, 3, 6, 13), then the routing pipeline that consumes them (14), then the surfaces (4, 5, 7, 8, 15, 16), then the polish that wraps them (9, 10, 11, 12).

Story 14 (Band A post-OTP routing) sits at the center. It needs the role model from 2, the matter-stage column from 3, the default-lead config from 13, and the welcome-message draft pipeline from 8. Everything downstream consumes a client record produced by 14.

## Sequencing rationale

Schema work clears the deck for surface work. Stories 2, 3, 6, 13 modify or add columns and tables that everything else reads from. Once those land, the routing pipeline (14) can produce a real client record from a Band A take, and stories 4, 5, 7, 8, 15, 16 can render against that record.

Stories 9 (notification batching), 10 (folder lock), 11 (mass message), and 12 (branded subdomain) are polish-and-scale features. They do not block the core flow, so they can run in parallel against the same client surfaces.

- All 16 stories closed with their own DoD satisfied.
- One end-to-end happy path: Band A lead lands via CaseLoad Screen, lawyer takes it, system creates client record at `matter_stage='intake'`, named-lead welcome draft surfaces in the lawyer's queue, lawyer reviews and sends, client receives magic-link invite, client lands on matter-stage home, internal chat thread exists alongside the client thread, lawyer can see the explainer library assigned to the matter, embed slot renders the firm's pre-authenticated URL.
- All migrations applied to Supabase project `qpzopweonveumvuqkqgw` (sequenced by the operator, not the dev session).
- No regressions: existing triage tests, firm-files tests, screen-engine sync check all pass.
- LSO Rule 4.2-1 audit: every client-facing string drafted in this epic clears the rule (no outcome promises, no specialist or expert language, no unverifiable superlatives).
- Doctrine register: four new DRs (DR-040 through DR-043) proposed in the architecture doc, ready for the operator to promote.

## Scope discipline

Phase 1 does not include: app folders in the sidebar, dynamic homepages keyed on custom-field rules, tasks with client visibility, recurring automations, the always-on form template editor, granular per-staff notification preferences, CSV import with field mapping, group/company channel UI (data model is built; UI exposes individual only), configurable client list columns, native e-signature, productized service storefronts, the auth bridge for embeds.

Phase 2 and Phase 3 candidates are documented in `S8-Phase1-architecture.md`. The operator decides scope for future phases after Phase 1 produces real usage signal.

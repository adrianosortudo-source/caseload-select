# S8.Phase1.07: Per-client internal team chat (right-rail)

**Status:** Shipped (2026-05-22)
**Assignee:** @dev
**Epic:** S8.Phase1
**Created:** 2026-05-18

## User narrative

The lawyer is on a call with the client and needs to ask the paralegal a question about the matter. Today that conversation happens off-platform (Slack, SMS, in-person). Phase 1 puts a persistent right-rail panel on every lawyer-facing matter surface: the client thread fills the main column, the internal thread fills the right rail. The two never cross. The internal thread is privileged work product.

## Acceptance criteria

- [ ] Right-rail panel renders on `/portal/[firmId]/clients/[matterId]` and on the screened-lead brief view `/portal/[firmId]/triage/[leadId]` (only when the lead has been taken and has a `matter_id`).
- [ ] Right rail is a fixed-width sticky panel on desktop; on mobile (below 1024px) it collapses to a tab toggle next to "Messages."
- [ ] The right-rail compose box submits messages with `channel_type='internal'`. The main-column compose box submits with `channel_type='client'`.
- [ ] Both panels render via the shared `MessageThread` React component, parameterized on `channel_type` for the filter and the default-write payload.
- [ ] Visibility: admin role sees both panels. Staff role sees both panels (subject to `canSeeMatter` from Story 2). Client role never sees the right-rail surface; client sessions cannot fetch internal messages via any API path.
- [ ] Internal messages do NOT trigger client-side notifications. They do trigger admin / staff notifications via the batched outbox (Story 9).
- [ ] No internal-thread content appears in any export endpoint, public-share endpoint, or webhook payload (extend the GHL outbound webhook helper to strip internal-thread metadata from `latest_message` payloads).

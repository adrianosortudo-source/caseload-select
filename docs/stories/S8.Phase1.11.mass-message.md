# S8.Phase1.11: Mass-message mode (one-to-many)

**Status:** Shipped (2026-05-22)
**Assignee:** @dev
**Epic:** S8.Phase1
**Created:** 2026-05-18

## User narrative

The lawyer needs to send the same message to many clients at once (office closure, holiday greeting, policy change) without exposing other recipients. Each client should see only their own thread copy. Without this, the firm falls back to BCC email blasts that hit spam filters and feel impersonal.

## Acceptance criteria

- [ ] New page `/portal/[firmId]/broadcast/page.tsx`: composer with: recipient picker (multi-select active matters), body textarea, audience filters ("All active matters", "By matter stage", "By practice area"), preview pane showing "Sending to N clients".
- [ ] New API route `POST /api/portal/[firmId]/messages/broadcast` body `{ recipient_matter_ids: string[], body: string }`. Inserts one source row into `matter_messages` with `broadcast_id = uuid()`, then inserts one per-recipient row in `matter_message_recipients` per matter, then fan-outs N child messages (one per matter, each carrying the source body, each linked back to the broadcast_id).
- [ ] Admin role only. Staff and clients cannot broadcast.
- [ ] Broadcast messages render in the lawyer-side message thread on each individual matter as if the lawyer had sent that message to that one client.
- [ ] No "Reply All" affordance from the client side; each recipient's thread is independent. A client reply lands on their thread only.
- [ ] Notifications go through the batched outbox (Story 9). A burst of 50 broadcast messages produces at most one digest per recipient in the next batch window.
- [ ] The broadcast composer surfaces a confirmation modal: "You are about to send this message to N clients. This action cannot be undone." Admin confirms before fan-out runs.

# S8.Phase1.01: Magic-link invite

**Status:** Shipped (2026-05-22)
**Assignee:** @dev
**Epic:** S8.Phase1
**Created:** 2026-05-18

## User narrative

When a lawyer takes a Band A screened lead and sends the welcome draft, the system needs to invite the client. The lawyer should be able to either let the platform send a branded invite email or copy the magic-link URL to paste into their own SMS or email thread that already exists with the client. Forcing the client through an unfamiliar "set a password" step kills adoption; the invite should land them in the portal with one click.

## Acceptance criteria

- [ ] New API route `POST /api/portal/[firmId]/matters/[matterId]/invite` returns a magic-link URL bound to `(firm_id, matter_id, recipient_email)` with 48-hour TTL.
- [ ] The token verifier accepts a new role value `client` and returns a session whose `matter_id` matches the URL path.
- [ ] `getMatterSession(matterId)` helper added alongside `getFirmSession(firmId)` for client-side route gating.
- [ ] The invite endpoint supports two modes: `mode='email'` sends a branded email via Resend; `mode='link-only'` returns the URL without sending.
- [ ] The welcome-draft-send action (Story 8) calls `mode='email'` by default and surfaces the link in the response so the lawyer can copy it.
- [ ] First-click flow: client clicks the magic link, lands on `/portal/[firmId]/m/[matterId]`, session cookie is set with role=`client`, no password prompt anywhere.
- [ ] Repeat clicks within the 48-hour TTL re-issue the session cookie. Expired clicks redirect to `/portal/login?error=expired-client&matter=...`.
- [ ] Google SSO is NOT in Phase 1 scope. Email magic-link only.

## Files to modify / create

- `src/app/api/portal/[firmId]/matters/[matterId]/invite/route.ts` (new)
- `src/lib/portal-auth.ts` (modify: add `client` role to `PortalRole`, add `matter_id` to `PortalSession`, add `getMatterSession()`)
- `src/app/api/portal/login/route.ts` (modify: handle client-role tokens, redirect to matter home)

## Schema changes

None. The token is HMAC-signed payload; no DB row needed. The `client_matters` table from Story 3 is the source of truth for matter ownership.

## Test plan

- vitest: `portal-auth.test.ts`: assert `client` role tokens validate, expired client tokens return null, matter_id mismatch returns null.
- vitest: `invite-route.test.ts`: assert `mode='email'` calls Resend with the right template, `mode='link-only'` does not call Resend.
- Manual: take a Band A lead → press Send → confirm Resend email arrives → click link → lands on matter home with no password.

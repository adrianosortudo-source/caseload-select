# S8.Phase1.10: Files folder-lock toggle

**Status:** Shipped (2026-05-22)
**Assignee:** @dev
**Epic:** S8.Phase1
**Created:** 2026-05-18

## User narrative

Legal matters have file taxonomies. A client deciding to tidy up their folders corrupts the firm's expectation of where things live. Phase 1 adds a per-firm toggle that controls whether the client can reorganize folders inside the portal file surface. Off by default. The lawyer sets up consistent folders ("Intake docs", "Signed retainer", "Court filings") and the client can fill them but cannot move them.

## Acceptance criteria

- [ ] New column `intake_firms.client_files_locked BOOLEAN DEFAULT false` on the routing-config migration (Story 13).
- [ ] Existing `firm_files` API routes extended: when the session is a client role and `client_files_locked = true` on the firm, write operations that mutate folder structure (rename, move, delete folders) return 403. File uploads into existing folders are still permitted.
- [ ] The client-side files page renders the toggle state as read-only chrome: folder names are not editable when locked.
- [ ] The lawyer-side firm-files page surfaces an inline switch labeled "Allow clients to reorganize folders" tied to `client_files_locked`. Admin role only.
- [ ] No new API route for the toggle itself; reuse the existing `/api/portal/[firmId]/admin/firm-settings` pattern (or add a small `PATCH /api/portal/[firmId]/admin/folder-lock` if no firm-settings route exists).
- [ ] Operator-curated default folder set seeded on matter creation: `Intake`, `Retainer`, `Correspondence`, `Filings`. Operator can change the seed list in `src/lib/default-folders.ts` once; Phase 2 covers per-firm customization.

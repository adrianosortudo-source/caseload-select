# S8.Phase1.02: Two-role permissions (admin / staff)

**Status:** Shipped (2026-05-22)
**Assignee:** @dev
**Epic:** S8.Phase1
**Created:** 2026-05-18

## User narrative

A solo lawyer running the firm needs to see every matter. A paralegal supporting that lawyer needs to see only the matters they are assigned to. The current role model is `lawyer | operator | client`; Phase 1 splits the firm-side role into `admin` and `staff` so visibility can be gated without building a permission matrix the firm will never use.

## Acceptance criteria

- [ ] `firm_lawyers.role` extended to accept `'admin'` and `'staff'` in addition to existing `'lawyer'` and `'operator'`. Existing `'lawyer'` rows treated as `'admin'` by the resolver (backward-compatible alias, no backfill required).
- [ ] `firm_lawyers` adds `display_name TEXT` and `title TEXT` columns (consumed by the welcome-draft template in Story 8).
- [ ] `PortalRole` type updated to `'admin' | 'staff' | 'operator' | 'client'`. The `lawyer` literal stays in `verifyPortalToken()` for backward compatibility and coerces to `admin`.
- [ ] New helper `canSeeMatter(session, matter): boolean`: admins see all firm matters, staff see only matters where `matter.lead_id === session.lawyer_id` OR `session.lawyer_id IN matter.assignee_ids`, operators see all, clients see only their own.
- [ ] Every existing portal route that reads matter or message data calls `canSeeMatter` before returning data.
- [ ] No admin / staff distinction surfaces in the UI in Phase 1 beyond the data-visibility filter. No role-management page. The operator sets `firm_lawyers.role` in the database during onboarding.

# S8.Phase1.03: Client matter-stage state machine

**Status:** Shipped (2026-05-22)
**Assignee:** @dev
**Epic:** S8.Phase1
**Created:** 2026-05-18

## User narrative

The portal has no concept of a signed matter today. Every action between "the lawyer takes a lead" and "the matter is closed" is invisible to the platform; the lawyer carries it in their head or in their PMS. Phase 1 introduces a five-stage state machine (`intake → retainer_pending → active → closing → closed`) so the system knows where each matter sits and the client homepage can show "what's next" based on stage.

## Acceptance criteria

- [ ] New table `client_matters` with columns documented in the migration plan (id, firm_id, source_screened_lead_id, lead_id, assignee_ids JSONB, matter_stage, matter_type, practice_area, primary_email, primary_phone, primary_name, welcome_draft_html, welcome_draft_sent_at, embed_url, created_at, updated_at).
- [ ] `matter_stage` CHECK constraint enforces the five-value enum.
- [ ] New table `matter_stage_events` (id, matter_id, from_stage, to_stage, actor_id, actor_role, note, created_at).
- [ ] Pure helper `validateStageTransition(from, to): boolean` returns true for legal transitions, false otherwise. Only forward transitions allowed.
- [ ] New API route `POST /api/portal/[firmId]/matters/[matterId]/stage` body `{ to: matter_stage, note?: string }`. Validates the transition, writes the row, writes the event, fires the appropriate journey cadence (J7 on `retainer_pending→active`, J9 on `active→closing`, J11+J12 on `closing→closed`).
- [ ] Take-action handler at `/api/portal/[firmId]/triage/[leadId]/take` extended: on take of a Band A screened lead, insert a new `client_matters` row at `matter_stage='intake'`, snapshot `default_lead_id` and `default_assignees` from firm config, draft the welcome message via `buildWelcomeDraft()` (Story 8), return the new `matter_id` in the response.
- [ ] Non-Band-A takes do NOT create a matter automatically in Phase 1.

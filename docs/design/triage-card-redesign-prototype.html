Triage Queue Card — Redesign Prototype▪

2026-05-26 · CaseLoad Select brand · static HTML preview

Hierarchy change + search affordance

The lawyer's daily mental model is "who reached out and when," not "what category of matter." Today's card leads with the matter-type label (SHAREHOLDER DISPUTE / WRONGFUL DISMISSAL), pushing the lead's name and arrival time into small metadata. Proposed redesign: name as primary, arrival timestamp as secondary, matter type and channel as tertiary tags. Because the queue is now scannable by person, the proposed view adds a queue header with a one-field search (name, phone, email, postal, lead ref, matter type) plus filter chips for band and channel.

Design notes for review

  • Band chip stays primary signal of priority. Bigger on the new card (56×56 vs 44×44) — anchors the row visually.
  • Name as the lead's identity. 22px Manrope 700, matches the brand header weight. The lawyer scans the queue by person; reading "Sarah Patel" first means they can recognise repeat callers, family connections, or referrals at a glance.
  • Contact right under the name, click-to-call and mailto links. The lawyer's natural action after picking a card is to call; one click should do it.
  • Arrival timestamp gets its own row with absolute + relative ("May 25, 2026 · 10:33 PM" · "2 hours ago"). Operator review feedback: response time is the single most important metric — surfacing it on the card lets lawyers triage stale leads at a glance.
  • Decision-window countdown moves top-right, separated from the name. "Decision in 24h 19m" reads as a clock, not a status word. The countdown turns red-ish when under 6h (handled by .clock.urgent class).
  • Matter type becomes a tag, not the headline. Same with channel. Lead ID drops to the right of the tag row in small Oxanium — useful for ops but not for the lawyer's first read.
  • Snapshot, fee estimate, and four-axis stay below a thin hairline separator. Body is still the diagnostic content, just no longer competing with the identity row.
  • Four-axis row: zero scores grey out (axis.zero class) so the lawyer's eye is drawn to the live signal. VAL/SMP/URG/RDY column ordering preserved (operator habit).
  • Square corners + parchment palette preserved. Border-radius 4px matches the brand book; no rounded card shapes.
  • Queue header adds a search field and filter chips. Today's portal has only a band filter as a URL param. With the NAP-first card, searching by lead name (or phone, email, postal, lead ref) becomes the natural way to find a specific person. Filter chips for band + channel sit below the search, mirroring the way operators already think about the queue.
  • Search is one input, multi-field match. One field, lower-cased substring match across name + phone + email + postal + lead ref + matter type. No advanced syntax, no separate fields. The lawyer types what they remember; the queue narrows.
  • Filter chip counts. Each band chip shows the lead count in that band (A 12, B 18, etc.). Operator sees workload distribution at a glance without scrolling.
  • Result count + reset. "Showing 2 of 47 · band A + B" reads back the active filters and total. A reset link clears search + filters in one click.

If approved — implementation scope

  • File to change: src/app/portal/[firmId]/triage/page.tsx (queue list rendering) and possibly a new components/portal/TriageQueueCard.tsx if we want to extract.
  • Data already in the screened_leads row: contact_name, contact_phone, contact_email, contact_postal_code (NAP), created_at (arrival timestamp), decision_deadline (countdown), band, matter_type, slot_answers.channel, brief_json.matter_snapshot, brief_json.fee_estimate, brief_json.four_axis.
  • Relative-time formatting helper: add a small formatRelativeArrival(created_at) in src/lib/decision-timer.ts or sibling. "Just now" / "2 hours ago" / "Yesterday at 10:33 PM" / "May 14, 2026" beyond a week.
  • Click targets: card stays clickable as a whole (navigates to brief view). Phone + email links inside the contact row should stopPropagation so they don't accidentally open the brief.
  • Mobile: the 3-column grid collapses to single column at <640px; band chip + name + countdown stack vertically.
  • Search + filter implementation: client-side filter is sufficient (queue page already loads all triaging rows). New components/portal/TriageQueueHeader.tsx owns the search input + filter chip state. Debounce the search input 100ms to keep keystroke latency low. Filter state syncs to URL params so the lawyer can bookmark a filtered view: ?q=patel&band=A,B&channel=messenger,whatsapp. The existing ?band= param keeps working (single-value form), with the comma-separated form supported on top.
  • Searchable fields (lower-cased substring match): contact_name, contact_phone (digits-only stripped before match, so "647-555-9999" finds "(647) 555 9999"), contact_email, contact_postal_code (whitespace stripped), lead_ref (the L-2026-05-26-Z3A short code), matter_type (matched against the human-readable label, not the enum string).
  • Chip counts computed once per render from the full queue array: const counts = { A: leads.filter(l => l.band === 'A').length, ... }. No extra fetch. Counts reflect the full queue, NOT the search-narrowed list (operator wants to see the workload, not the search result distribution).
  • Empty state when search returns zero matches: "No leads match 'patel' in this view. to see all 47."
  • Out of scope for v1: language filter, practice-area filter, date range. Matter-type filter is technically possible but the matter-type list is long; defer until the queue grows large enough to need it.
  • No engine changes. No sandbox sync. Pure portal UX.

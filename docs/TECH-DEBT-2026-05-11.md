## 1. Host-side file truncation when the agent writes via Edit / Write tools

**Observed.** Two confirmed instances in this session:

1. **`CaseLoadScreen_2.0_2026-05-03/src/brief-render.ts`** — three trailing null bytes (`\x00\x00\x00`) appended at EOF after a sequence of Edit-tool operations. Caught by `tsc` with `error TS1127: Invalid character` at line 652, columns 1-3 (one column per null byte). Fixed by stripping trailing nulls via Python and re-writing with a single trailing newline.

2. **Sandbox `package.json`** — file truncated mid-write at byte 456, halfway through a dependency entry (`"@vercel/node": "^5.7.15",\n    "` then nothing). `npm install` failed with `JSON.parse Unterminated string in JSON at position 456`. The parallel-session linter had been adding `franc` to dependencies; the write didn't complete. Fixed by writing a fresh valid `package.json` via bash heredoc.

A separate but related instance happened to `Version3_CaseLoadSelect/CaseLoad_Select_CRM_Bible_v5.1.md` and `05_Product/caseload-select-app/CLAUDE.md` earlier in the session — both truncated mid-content during Edit operations. Recovered via bash heredoc / Python rewrites. The MD recovery was successful; the app CLAUDE.md brand-assets table tail remained truncated and was capped with a recovery-note HTML comment for later git restore.

**Pattern.** Three independent files truncated in the same session. Each lost trailing content of varying size (3 bytes to ~50 lines). The pattern looks like a buffered write that didn't flush its final chunk before the next operation observed the file.

**Hypothesis worth checking.** If the agent's host runs the file I/O through a WSL/Windows bridge (the agent's sandbox is Linux x86_64, the user's machine is Windows; the workspace lives on the Windows side at `D:\`), the bridge may drop trailing bytes on certain file sizes or content shapes. The truncations happened at byte boundaries that look suspicious: 456 (package.json), 23,233 minus 3 trailing nulls (brief-render.ts). Worth dumping the host bridge's buffer behaviour or running a fuzz of "write file of size N, verify N bytes landed" across a range of N values.

**Mitigation in the meantime.** Every file the agent writes via Edit/Write needs immediate verification:

```bash
# After every Edit or Write to a file we care about:
F=/path/to/file
wc -c "$F"                                  # size sanity
tail -c 10 "$F" | od -c | head -1           # check for trailing nulls
test "$(tail -c 1 "$F" | od -An -c | tr -d ' ')" = "\\n" && echo OK || echo "missing trailing newline"
# For JSON / structured files:
python3 -c "import json; json.load(open('$F'))"
# For TypeScript / source files, periodically run tsc to catch invalid characters
```

The agent's sync verification script (`scripts/check-engine-sync.sh`) catches engine-port drift but does not catch trailing-null corruption inside files that both sides duplicate. Add a "no trailing nulls" check to that script or a sibling pre-commit hook.

## 2. Sandbox npm cannot install platform-correct rollup native binary

**Observed.** Three install attempts in the agent's Linux x86_64 sandbox:

1. `npm install --include=optional` (lockfile freshly regenerated): 19 of 21+ rollup platform variants landed; `@rollup/rollup-linux-x64-gnu` and `@rollup/rollup-linux-x64-musl` were both missing. Confirmed npm/cli#4828.
2. `npm install --include=optional --force`: regression. Only `@rollup/pluginutils`, `@rollup/rollup-win32-x64-gnu`, `@rollup/rollup-win32-x64-msvc` landed. Even the main `rollup` package was removed from `node_modules`. The `--force` flag actively made things worse.
3. `rm -rf node_modules && npm install --include=optional` (keeping the lockfile): same as attempt 2's regression. Only the three Windows variants present. The post-`--force` lockfile is what npm reads on the retry, and the regression sticks.

`npm rebuild rollup` as the surgical fallback then failed with `EPERM: operation not permitted, unlink '.../node_modules/.bin/rollup'`. The sandbox filesystem (mounted at `/sessions/eager-elegant-volta/mnt/`) appears to disallow symlink unlink, which is what npm uses to swap binary links during a rebuild.

**Result.** `npx vitest run`, `npx vite build`, and the dev server cannot run in this sandbox. The voice-intake curl smoke tests cannot be executed here. The Vercel deploy target may or may not hit the same issue; Vercel's build environment differs from this sandbox.

**Hypothesis worth checking.** The mounted filesystem at `/sessions/eager-elegant-volta/mnt/` may have noexec or no-symlink-rewrite semantics that npm trips over. The agent's previous successful runs (e.g., `tsc --noEmit` which only reads) suggest reads work fine; writes that involve symlinks or specific permission ops break. If the sandbox is intended to support the full Node ecosystem, that mount needs full rwx + symlink support.

**Mitigation in the meantime.** Run the install + tests + build on the operator's local machine (Windows or any non-restricted Linux), not in the agent's sandbox. The lockfile is committable from either side; the install state is per-host and discarded.

**Once Vercel's deploy validates the install path:** the agent's sandbox can do typecheck-only verification (which works), the operator's machine or Vercel handles the full install + test + build.
